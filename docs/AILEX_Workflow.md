**AILEX — Памятка по рабочему циклу**

*Как мы вносим правки в код и обновляем приложение*

# 0. Инструкция для нового чата (читать первым делом)

При каждом новом чате Claude должен первым делом прочитать документацию проекта с GitHub. Без этого Claude работает вслепую и может предложить то, что уже реализовано.

**Команды для чтения документов в начале сессии:**

```
# 1) Текущее состояние — читать первым. Заодно отсюда берём имя актуальной спеки.
curl -s https://raw.githubusercontent.com/4wood70/ailex/main/docs/AILEX_Status.md -o /tmp/ailex_status.md
cat /tmp/ailex_status.md

# 2) Последняя спецификация. Имя версионируется (AILEX_4_19 → 4_22 → …), поэтому НЕ хардкодим:
#    Status в шапке несёт строку «Текущая спецификация: docs/AILEX_X_XX.md» — читаем её оттуда.
#    (Сознательно НЕ используем GitHub API /contents — у него лимит 60 запросов/час на общий IP, падает с 403.)
SPEC=$(grep -oP 'Текущая спецификация: docs/\KAILEX_\d+_\d+\.md' /tmp/ailex_status.md)
curl -s "https://raw.githubusercontent.com/4wood70/ailex/main/docs/${SPEC:-AILEX_4_22.md}"

# 3) Глоссарий и принятые решения
curl -s https://raw.githubusercontent.com/4wood70/ailex/main/docs/AILEX_Glossary.md
```

Порядок чтения: сначала `AILEX_Status.md` (текущее состояние + указатель спеки), затем спека и глоссарий по необходимости. Указатель спеки в Status обновляется при каждом подъёме версии спеки (см. раздел 9), поэтому всегда ведёт на актуальный файл; `AILEX_4_22.md` в команде — лишь страховочное значение по умолчанию, если строку-указатель не нашли.

**Важно:** сам `AILEX_Workflow.md` (этот файл) живёт в документах проекта Claude и подгружается в контекст автоматически — отдельной `curl`-командой его читать НЕ нужно, и в git-репозитории его нет. Раздел 0 запускает чтение остального; читать им самого себя из git было бы кольцом (и вернуло бы 404).

Если Constantin говорит «прочитай документацию» или «освежи контекст» — выполнить эти три команды (Status → последняя спека → глоссарий).

**Текущий билд:** проверять через:
```
curl -s https://raw.githubusercontent.com/4wood70/ailex/main/index.html | grep APP_VERSION
```

# 1. Структура репозитория

```
4wood70/ailex/
├── index.html          — основное приложение (вся логика, UI)
├── sw.js               — сервис-воркер (версионирование кэша)
├── manifest.webmanifest
├── icons/
└── docs/
    ├── AILEX_4_NN.md   — методическая конституция (спецификация); имя версионируется,
    │                     последняя на момент написания — AILEX_4_22.md
    ├── AILEX_Status.md — текущий статус, очередь задач
    └── AILEX_Glossary.md — глоссарий и принятые решения
```

`AILEX_Workflow.md` (этот файл) в git НЕ хранится — он живёт в документах проекта Claude и подгружается в контекст автоматически.

# 2. Нумерация версий

Два независимых счётчика:

**Билд** (`v35`, `v36`...) — поднимается при каждом коммите в GitHub. Прописывается в `APP_VERSION` (index.html) и `CACHE` (sw.js) — всегда совпадают. Проставляет Claude автоматически.

**Спецификация** (`4.19`, `4.20`...) — поднимается только при содержательном изменении: новая функция, изменение методики, новый контракт с Gemini, архитектурное решение. Мелкий багфикс или косметика — не повод. Название файла `AILEX_X_XX.md` всегда совпадает с последней записью в истории версий внутри документа.

| Что изменилось | Билд | Спецификация |
|---|---|---|
| Любой коммит (багфикс, косметика) | +1 | не меняется |
| Новая функция / методика / контракт | +1 | +0.1 |

# 3. Роли (кто что делает)

- **Claude** — готовит код: пишет/правит файлы целиком, сам проставляет номер билда. Код вручную не читается и не редактируется.
- **Constantin** — кладёт готовые файлы в репозиторий через GitHub Desktop и делает коммит.
- **Claude НЕ может коммитить сам** — последний шаг «положить файл на GitHub» всегда за Константином.

# 4. Полный цикл одной правки

1. **Обсуждаем.** Constantin говорит что нужно изменить. Решаем вместе как именно.
2. **Обновляем спецификацию** (если изменение содержательное). Claude готовит обновлённый `AILEX_X_XX.md` и `AILEX_Status.md`.
3. **Claude готовит файлы.** Получаете целиком готовые `index.html` и/или `sw.js` с поднятым билдом. Документы — если нужно.
4. **Заменяете файлы через GitHub Desktop.** Скопировать в локальную папку репозитория → commit → push.
5. **Сверяемся.** Claude читает файлы с GitHub и подтверждает что правка легла верно.

# 5. Проверка после коммита

```bash
# Проверить билд
curl -s https://raw.githubusercontent.com/4wood70/ailex/main/index.html | grep APP_VERSION
curl -s https://raw.githubusercontent.com/4wood70/ailex/main/sw.js | grep CACHE

# Проверить конкретный фрагмент кода
curl -s https://raw.githubusercontent.com/4wood70/ailex/main/index.html | grep -A3 "const LANGS"
```

# 6. Сообщения коммитов (GitHub Desktop)

**Где выдаётся коммит-сообщение:** Claude всегда выдаёт готовое сообщение коммита **прямо в чат, отдельным блоком кода для копирования** — НЕ вписывает его в этот документ и не хранит примеры коммитов здесь. Документ описывает только формат; конкретные сообщения живут в чате той сессии, где сделан билд.

**Язык:** сообщения коммитов пишутся **строго на английском языке**.

**Формат** — два поля, Summary и Description. Claude выдаёт их в чате одним блоком вида:

```
vNN: short imperative summary of the change

- bullet describing change 1
- bullet describing change 2
- rationale where it matters
```

Правила содержания:
- Summary — одна строка, начинается с номера билда (`vNN:`), повелительное наклонение, по сути изменения.
- Description — маркированный список конкретных правок; где важно — краткое обоснование (почему так, а не иначе).
- Для чисто документальных коммитов summary вида `docs: update Workflow and Status` (без номера билда).

# 7. Два железных правила

**Правило 1. После каждого коммита — проверить билд.**
Claude читает `APP_VERSION` с GitHub и сравнивает с тем что было отправлено.

**Правило 2. Билд поднимаем всегда — даже для крошечной правки.**
`APP_VERSION` в index.html и `CACHE` в sw.js должны совпадать (например `v35` и `ailex-v35`). Без этого у пользователей залипает старый кэш PWA.

# 8. Если что-то пошло не так

- Claude видит старый код → попросить прочитать файл заново с GitHub (команды из раздела 0).
- Обновление не видно в приложении → проверить что билд поднят и коммит прошёл.
- Не уверены какая версия в репозитории → `curl` на `APP_VERSION` (см. раздел 5).

# 9. Завершение сессии — обязательный шаг

**После того как готовые файлы выданы Constantin'у, но ДО закрытия чата:**

Если в сессии были содержательные изменения (новые функции, багфиксы, архитектурные решения) — Claude обязан подготовить обновлённый `AILEX_Status.md` и предложить его Constantin'у.

**Что включать в обновление Status:**
- Все реализованные в этой сессии изменения — добавить в раздел 1 с указанием билда.
- Очередь задач (раздел 7) — очистить то, что реализовано.
- Текущий фокус (раздел 6) — обновить.
- Дата последнего обновления — проставить текущую.
- Указатель спеки в шапке («Текущая спецификация: docs/AILEX_X_XX.md») — обновить, если в сессии поднималась версия спеки. Раздел 0 инициализации читает имя актуальной спеки именно отсюда, поэтому строка обязана вести на свежий файл.

**Регулярная очистка списка приоритетов (обязательно каждую сессию):** раздел 7 (очередь) и раздел 6 (фокус) — это списки ТОЛЬКО актуального. Любой пункт, реализованный в коде, в ту же сессию убирается из развёрнутого вида: его место — короткая строка-сводка («QUEUE-NN — закрыт в vN, детали в разделе 1»), а полное описание уже живёт в разделе 1 (по билдам) и в истории git. Накапливать закрытые пункты в очереди запрещено — раздутый список заставляет следующую сессию тратить внимание на разбор давно сделанного и повышает риск предложить как новое то, что уже есть. Принцип: в разделах 6 и 7 не должно оставаться ни одного пункта, который полностью закрыт кодом. Пункты «mitigated подсказками» (сознательно без кода) и «проверить на тесте» (код не требуется) — это НЕ закрытые, их оставляем в свёрнутом виде с пометкой статуса.

**Почему это важно:** Claude не имеет памяти между сессиями и полностью опирается на документы. Если Status не обновлён после сессии — следующий чат начнётся с устаревшей картиной мира и может предложить то, что уже сделано, или потерять контекст принятых решений.

**Триггер для Constantin'а:** если в конце сессии Claude не предложил обновить документы — напомни: «обнови статус».

**Проверка соседних чатов:** если Constantin говорит что несколько последних сессий не документировались — Claude должен прочитать последние чаты проекта через инструмент `recent_chats` и восстановить пропущенные изменения в Status перед продолжением работы.

# 10. Проверка моделей Gemini (периодически)

Список моделей в `index.html` (`Gemini.MODELS`, ~строка 573) захардкожен сознательно — это модели, которым мы доверяем методику. Google периодически выкатывает новые flash и отключает старые. Автоподхват новейших отклонён (v44 + обсуждение v57): эндпоинт ListModels не отдаёт лимиты, «новее» ≠ «лучше для нас», новая модель может тихо сдвинуть методику (а конституция требует: смена версии Gemini не меняет методику). Поэтому проверка — ручная, по триггеру, человек в цикле.

**Когда проверять:**
- Загрузка приложения стала медленной (10+ сек) — первый признак мёртвой модели в списке (симптом v44: залипший дохлый эндпоинт).
- Планово раз в ~1–2 месяца.
- Когда узнал о новой версии Gemini flash.

**Что живо у ключа:**
```
curl -s "https://generativelanguage.googleapis.com/v1beta/models?key=ВАШ_КЛЮЧ" | grep '"name"'
```
Возвращает модели, реально доступные ключу сейчас. Если модель из нашего `MODELS` тут не появляется — она мёртвая, убрать из кода.

**Лимиты API НЕ отдаёт** — RPM / TPM / RPD смотреть вручную в документации Google (страница Rate limits / Pricing, Free Tier). Порядок перебора в `MODELS` построен на RPM (щедрые лимиты — первыми; узкое место — именно RPM, не объём токенов), поэтому лимиты новой модели надо узнать ДО добавления.

**Что делать по итогу:**
- Мёртвую модель — удалить из `MODELS`, поднять билд (иначе залипший эндпоинт → медленная загрузка).
- Новую flash — НЕ добавлять автоматически. Принести в чат, обсудить, прогнать тест (держит ли строгий JSON-контракт, не поехала ли методика/тон), и только тогда добавить в `MODELS` на позицию по её RPM. Поднять билд.
- Известный EOL: `gemini-2.5-flash` — октябрь 2026, удалить заранее.

**Отклонённые автоварианты (не возвращаться без причины):** (1) алиас `gemini-flash-latest` — непрозрачно, какая модель реально отвечает, хуже для отладки методики; (2) автоподхват новейшей из ListModels — нет лимитов в ответе, хрупкая сортировка по «версии», риск тихого дрейфа методики.
